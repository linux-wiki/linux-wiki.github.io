  • 内存子系统

Linux 内存管理 不是设计一个严格分层架构，而是：把“内存这件事”拆成多个独立子系统，每个解决一个问题，然后通过 struct page / VMA / folio 串起来

Linux内存管理拆分方法

1. 按“抽象层级”拆分（虚拟→物理）

用户进程
→ VMA / mm_struct
→ page table
→ struct page
→ zone/node
→ MMU

对应子系统：

层级子系统代码
虚拟地址mm/mmap.cVMA
地址翻译arch/mmpage table
物理页mm/page_alloc.cbuddy
硬件arch/x86/mmMMU

👉 这一层解决的是：

“地址怎么从虚拟变成物理”

2. 按“内存用途”拆（非常关键）

Linux 把“用内存干什么”拆成不同系统：

2.1 页缓存（Page Cache）

文件 IO 用的内存

归属：

mm/filemap.c

结构：

inode → address_space → folio/page

特点：

  • IO缓存
  • 回写（writeback）
  • LRU管理

2.2 slab / slub

小对象分配器

归属：

mm/slub.c

用途：

  • task_struct
  • dentry
  • inode
  • socket

特点：

page → slab → object

👉 本质：

page 之上再切一层“对象管理”

2.3 vmalloc

虚拟连续但物理不连续

归属：

特点：

  • 不走 buddy 连续分配
  • 只管虚拟映射

👉 这一层拆分逻辑是：

“不同使用方式 → 不同 allocator”

3.3 struct page（统一描述符）

每一页物理内存的统一抽象

关键：

👉 作用：

所有上层系统最终都落到 page

4. 按“分配方式”拆（Buddy系统）

buddy 是底层物理页分配器：

它只干一件事：

给上层提供 2^n 页连续内存

buddy 不关心：

  • 文件
  • slab
  • VMA
  • cache

它只关心：

page free / split / merge

关键理解

如果用 API 依赖总结 Linux MM：

核心 API 只有三个入口

mmap / fault / alloc_pages

所有系统最终依赖一个API

alloc_pages()

所有对象最终依赖一个结构

struct page

slab / cache / vmalloc 都是

alloc_pages + 不同封装
