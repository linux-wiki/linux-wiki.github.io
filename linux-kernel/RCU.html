RCU是什么

RCU（Read-Copy-Update）是一种面向读多写少场景的并发同步机制：

读操作几乎无锁，写操作通过“复制 + 延迟回收”保证一致性

它的核心思想不是“阻止冲突”，而是：

  • 允许读线程并发读旧数据
  • 写线程生成新版本
  • 旧版本延迟释放，等所有读者结束后再回收

⚠️ 边界很重要

RCU 解决的是：

  • 读多写少的数据一致性问题

RCU 不适合：

  • 强一致写写互斥
  • 需要实时看到最新值的逻辑

RCU 解决什么问题

在没有 RCU 之前，内核常见同步方式：

方法问题
spinlock读也被阻塞，性能差
rwlock读可以并发，但写仍然阻塞读
mutex太重

典型痛点场景

例如：

  • dentry 缓存
  • inode 查找
  • 路由表
  • Netfilter hook
  • task struct 访问

这些特点是：

  • 读远多于写（99% 读）
  • 读路径必须极快
  • 不能加锁影响性能

👉 所以 RCU 出现：

把“同步问题”变成“版本问题”

RCU 的核心机制

RCU 由三部分组成：

1. 读侧（RCU Reader）

特点：

  • 无锁
  • 通过 rcu_read_lock() 标记临界区
  • 访问数据时用 rcu_dereference()

核心规则：

读者不会阻塞写者，也不会被写者阻塞

示例逻辑：

rcu_read_lock();p = rcu_dereference(global_ptr);use(p);rcu_read_unlock();

👉 读者看到的是：

  • 当前版本 or 旧版本（但一定合法）

2. 写侧（RCU Writer）

写操作不是“修改”，而是：

❗复制 + 替换指针

流程：

  1. 复制旧对象
  2. 修改新对象
  3. 用 rcu_assign_pointer() 原子替换
  4. 等待旧读者结束
  5. 释放旧对象

3. 回收机制（Grace Period）

关键点：

旧数据不能立即释放

必须等待：

  • 所有 CPU 离开 RCU 读临界区

这个等待周期叫： Grace Period

RCU vs 其他机制（对比层）

机制读性能写性能一致性适用
spinlock差中强小临界区
rwlock中差强读多写少
RCU极强中/差最终一致超高读场景

关键本质区别：

spinlock/rwlock：

“阻止冲突”

RCU：

“允许冲突 + 版本隔离”

进阶理解（机制本质）

RCU 的本质其实是：

延迟回收 + 读无锁 + 写版本替换

可以抽象成：

旧版本 ----------\ 
              -> 并发读
新版本 ----------/

等“所有读者结束”后：

释放旧版本

用一句话总结

RCU 本质就是：

用“时间换空间一致性”，用“延迟回收”换“读路径零锁”
